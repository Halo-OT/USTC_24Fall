#### 2. **The Core Algorithm**

在本实验中，我们实现了时间多线程调度，并且通过上下文切换来实现两个程序（程序 A 和程序 B）在 LC-3 计算机上的轮流执行。主要算法涉及以下几部分：

1. **中断设置**：启用键盘中断和 TRAP x77。
2. **键盘中断处理程序**：检查是否按下了 `Q` 键，如果按下 `Q`，则停止执行程序。
3. **TRAP x77 处理程序**：程序 A 或程序 B 执行时会调用 `TRAP x77`，触发程序切换。
4. **上下文切换**：当程序 A 和程序 B 之间发生切换时，保存当前程序的上下文（包括寄存器和程序计数器），并加载下一个程序的上下文。

程序的执行流程是：程序 A 执行输出一条消息，调用 `TRAP x77` 请求切换到程序 B；程序 B 执行输出另一条消息，调用 `TRAP x77` 请求切换回程序 A，直到用户按下 `Q` 键。

#### 3. **Critical Code to Implement the Algorithm**

在实验中，以下代码部分是实现多线程调度的关键：

- **启用键盘中断**：

```asm
; Setup program
.ORIG x0800
; ==============================
; TODO

;Enable keyboard interrupt
LDI R0, KBSR
LD R1, KBSRMask
NOT R0, R0
NOT R1, R1
AND R0, R0, R1
NOT R0, R0
;R0 = KBSR OR x4000
STI R0, KBSR
;KBSR[14] = 1

;Setup ISR for keyboard
LD R0, KBISRAddr
STI R0, KBVecAddr

;Setup handler for trap
LD R0, TrapISRAddr
STI R0, TrapVecAddr

;Calls user program A
LD R0, PSR_A
ADD R6, R6, x-1
STR R0, R6, x0
;Push PSR of program A into the stack
LD R0, PC_A
ADD R6, R6, x-1
STR R0, R6, x0
;Push PC of program A into the stack
RTI
; ==============================

KBSR .FILL xFE00
KBSRMask .FILL x4000
KBVecAddr .FILL x0180
KBISRAddr .FILL x1000
TrapVecAddr .FILL x0077
TrapISRAddr .FILL x1800
PSR_A .FILL x8002
PC_A .FILL x4090

.END
```

该代码段通过设置 KBSR 寄存器的第 14 位来启用键盘中断，使得程序能够在按下键盘时响应。

- **TRAP x77 处理程序**：

```asm
; Trap x77 handler
.ORIG x1800
; ==============================
; TODO
;Call switch program
LD R7, SwitchProgramAddr
JMP R7
; ==============================

SwitchProgramAddr .FILL x2000
.END
```

通过触发 TRAP x77，调用 `SwitchProgram` 来切换程序 A 和程序 B。

- **上下文保存和恢复**：

```asm
STR R0, R7, x0
STR R1, R7, x1
...
LDR R0, R7, x0
LDR R1, R7, x1
```

在上下文切换时，通过 `STR` 和 `LDR` 指令保存和恢复寄存器的状态。

#### 6. **Traps and Pitfalls You've Found**

在研究程序的过程中，我发现了以下几个潜在的问题：

1. **寄存器值的保存与恢复**：在进行上下文切换时，必须保证每个寄存器（除了 `R7`）都被正确地保存和恢复。如果某个寄存器在上下文切换中被修改，程序将无法正确恢复其状态，可能导致崩溃。
2. **程序崩溃的可能性**：如果在键盘 ISR 中快速按下其他键（除了 `Q`），LC-3 可能会崩溃。这是因为程序没有适当的处理机制来防止在中断处理中对数据的并发访问。
3. **程序切换的复杂性**：程序切换部分的代码较为冗长（约 100 行），且包含大量的寄存器操作，这使得整个程序切换过程变得较为复杂且容易出错。

### 问题回答

#### 1. Talk about how you understand that we must not change any register in the ISR.

在 ISR（中断服务程序）中，我们必须避免改变除 `R7` 外的寄存器值。这是因为程序在进入中断服务程序时，其上下文（即寄存器的状态）是由操作系统或硬件保存的。在 ISR 中，我们通过修改 `R7` 寄存器来进行程序切换，但其他寄存器的值必须保留不变，以便在程序切换回来时，能够恢复原有的状态。如果不小心修改了其他寄存器的值，会导致恢复上下文时出错，从而使程序无法正确恢复执行。

#### 2. Using the program above, you may notice that if you hit some key (other than Q) quickly, LC-3 will crash. Suggest some possible cause of this.

可能的原因是：在键盘中断服务程序（ISR）中，程序快速响应按键时，可能发生了并发访问或寄存器的未保存状态。由于程序没有对键盘输入进行适当的检查或同步，当按键输入过于频繁时，ISR 可能会干扰到主程序的执行，导致程序崩溃。因此，应该增加对输入的检查和防止过多并发访问的机制。

#### 3. Implementing temporal multithreading is cumbersome in terms of scheduling, while for simultaneous multithreading things will be tricky when managing data synchronization. You may have also noticed that the program switch is very long (~100 lines). Paying such a high price, why computers of nowadays still implement multithreading?

现代计算机实现多线程的原因主要有两个：

1. **提高资源利用率**：通过时间分片调度（如本实验中的实现），即使只有单个处理器，计算机也可以在多个任务之间快速切换，从而提高 CPU 的利用率。这对于高效的系统管理和任务执行是必要的，尤其是在任务繁重的情况下。
2. **响应性和并发处理**：现代计算机需要在多个任务间进行高效的调度，以保证系统的响应性和同时处理多个用户请求或计算任务。虽然操作系统必须处理程序切换的复杂性和开销，但多线程调度使得计算机可以在多个任务之间灵活切换，从而提供更高的响应速度和计算能力。

尽管程序切换过程较长，但它对于提升系统的并发处理能力和多任务处理能力是必要的。现代计算机中的多线程通常是通过硬件（如多核处理器）和操作系统调度机制来实现的，以有效地管理大量的线程。
